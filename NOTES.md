PROJECT HANDOFF: "World's End" Fabric mod (worlds-end), continued from a previous chat.
What it does: After the Ender Dragon is defeated (detected by polling EndDragonFight#hasPreviouslyKilledDragon() on the End dimension each server tick — the Fabric AFTER_DEATH event does NOT fire for the dragon), a circular border centered on world spawn shrinks from the world edge (radius 30,000,000) toward zero. Blocks outside the circle are deleted. Deletions near players spawn a cosmetic DriftingBlockEntity that floats away radially, tilted 30° upward, 5 blocks at full scale then shrinking to zero over 5 more (40 ticks total), with budgeted block-break sounds. Per-player boss bars show "The End is X blocks away." A /worldsend start command exists for debugging.
Environment: Minecraft 26.2, Fabric Loader 0.19.3, Fabric API 0.154.0+26.2, Loom 1.17, Gradle 9.5.1, JDK 25 (OpenJDK). Mod ID is worlds-end (hyphen). Critical: this version uses Mojang's unobfuscated names, not classic Mojmap — several classes were renamed (ResourceLocation → Identifier but keeping fromNamespaceAndPath; EndDragonFight → EnderDragonFight; getSharedSpawnPos() is gone → level.getRespawnData().pos(); ChunkPos.x/z fields → x()/z() methods; toLong() gone). When an API name is uncertain, don't guess repeatedly — the game source is readable via Ctrl+click in VS Code, and the user is comfortable pasting vanilla classes on request; that workflow solved every naming problem.
Architecture (ShrinkController, three layers):

Ring sweep — each tick, clears the ring between the current radius and the next scheduled step. Candidate chunks are discovered by scanning around each player (never by scanning the circle's bounding box — that's O(r²) and lethal at radius 30M). Columns are sorted outermost-first, cleared under RING_BUDGET_PER_TICK (8192). Granular pacing: if the budget runs out mid-band, the radius advances to the outermost uncleared column's distance — never freeze-then-lurch (an earlier full-band-or-nothing rule froze the boss bar and caused a west-to-east planar sweep bug before the sort existed).
Wipe queue — deletion is lazy/"virtual": the radius is just a persisted number; chunks outside it are wiped when they become observable. Fed by CHUNK_LOAD, by FULL_CHUNK_STATUS_CHANGE (covers a documented gap where CHUNK_LOAD doesn't fire), and by a periodic audit (every 60 ticks, all loaded chunks around players not fully inside the circle are queued). Wipes skip all-air sections (hasOnlyAir()) so re-checking clean chunks is nearly free. No region-file surgery — deliberately rejected as risky and unnecessary.
Speed schedule — EndOfWorldState.shrinkPerTick(radius) table: 500 b/t above 1M, 50 above 100k, 5 above 10k, 0.5 above 2k, 0.05 endgame. Radius-only SavedData (codec-based SavedDataType), survives restarts.

Animation/visuals decisions (hard-won):

Animations select for visibility: nearest player within 64 blocks horizontal, ±16 vertical of their Y (never waste budget at bedrock), max ANIMATIONS_PER_COLUMN = 3 randomly sampled per column so the budget (96/tick) spreads across the whole face instead of clustering (greedy allocation caused big bare sections).
Entity pool uses a plain counter with backpressure (full pool = skip new spawns, MAX_LIVE = 1024). Two dead ends: cull-oldest (blocks popped mid-flight, looked bad) and a reference-holding deque (jammed on stale entities across world reloads). Counter decrements via Fabric's ServerEntityEvents.ENTITY_UNLOAD — Entity#setRemoved is final, can't override.
SOUND_BUDGET_PER_TICK = 64 is the user's tested sweet spot (verify the code has 64, not 8).
Renderer uses the 26.x submit pipeline (submit() + SubmitNodeCollector.submitMovingBlock), with a render state extending FallingBlockRenderState; modeled on vanilla FallingBlockRenderer. The old render()/MultiBufferSource/BlockRenderDispatcher path no longer exists.

Critical lesson — static state: all JVM-static state (queues, boss bar map, animation counter) leaks across singleplayer world reloads. Everything is reset via ServerLifecycleEvents.SERVER_STARTED → ShrinkController.reset() + DriftingBlockEntity.resetPool(), and boss bars re-bind when the ServerPlayer instance changes (!bar.getPlayers().contains(player) → re-add). Any new feature with static state must join this reset path.
